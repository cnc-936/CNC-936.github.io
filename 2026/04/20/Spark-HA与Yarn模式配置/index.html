Spark HA与Yarn模式配置

一、实验目的

  1. 理解Spark Standalone模式单点故障问题，掌握基于ZooKeeper实现Spark HA（高可用）的原理。
  2. 完成Spark Standalone HA集群的配置，实现Master节点的主备切换，解决单点故障。
  3. 掌握Spark On YARN模式的部署配置方法，实现Spark与Hadoop生态的整合。
  4. 理解YARN作为资源管理器的优势，会在YARN-client和YARN-cluster模式下提交Spark任务。
  5. 对比Spark Standalone HA模式与YARN模式的资源调度方式，明确两种模式的适用场景。

二、实验环境

2.1 基础环境

  • 已完成Spark Standalone 1主2从集群部署（192.168.10.101/102/103）；
  • 已完成ZooKeeper 3.8.0集群部署（3节点，IP：192.168.10.101/102/103，端口2181）；
  • 已完成Hadoop 3.3.6集群部署（1主2从，YARN资源管理器地址：192.168.10.101:8088）；
  • 所有节点JDK、Scala、Spark环境配置一致，SSH免密登录生效。

2.2 核心组件版本匹配

组件 版本 说明
Spark 3.5.0-bin-hadoop3 与Hadoop 3.3.6完美兼容
Hadoop 3.3.6 提供YARN资源调度和HDFS存储
ZooKeeper 3.8.0 实现Spark Master主备选举
JDK 1.8.0_301 所有组件基础运行依赖

三、实验步骤

1. Spark Standalone HA模式配置（基于ZooKeeper）

Spark Standalone默认单Master节点，存在单点故障问题（Master宕机则整个集群不可用）。通过ZooKeeper实现Master节点的主备选举，当Active Master宕机时，Standby Master自动切换为Active，保证集群持续可用，这就是Spark HA模式。

1.1 核心配置原理

  • ZooKeeper作为分布式协调服务，存储Spark集群的元数据和Master节点状态；
  • 部署多个Master节点（1主1备/多备），ZooKeeper通过选举机制确定Active Master；
  • Worker节点和Driver端通过ZooKeeper获取当前Active Master地址，实现故障自动感知。

1.2 修改spark-env.sh配置文件（所有节点）

进入Spark配置目录

cd $SPARK_HOME/conf

编辑spark-env.sh文件

vim spark-env.sh

保留原有配置，添加以下HA核心配置

export SPARK_DAEMON_JAVA_OPTS=”-Dspark.deploy.recoveryMode=ZOOKEEPER
-Dspark.deploy.zookeeper.url=192.168.10.101:2181,192.168.10.102:2181,192.168.10.103:2181
-Dspark.deploy.zookeeper.dir=/spark-ha”

说明：

recoveryMode=ZOOKEEPER：指定恢复模式为ZooKeeper

zookeeper.url：ZooKeeper集群地址

zookeeper.dir：Spark在ZooKeeper中的元数据存储目录

1.3 分发配置到所有节点

分发spark-env.sh到Worker1

scp $SPARK_HOME/conf/spark-env.sh root@192.168.10.102:$SPARK_HOME/conf/

分发spark-env.sh到Worker2

scp $SPARK_HOME/conf/spark-env.sh root@192.168.10.103:$SPARK_HOME/conf/
1.4 启动 ZooKeeper 集群

在所有ZooKeeper节点执行启动命令

zkServer.sh start

查看ZooKeeper状态（确保所有节点为follow/leader）

zkServer.sh status
1.5 启动 Spark HA 集群
启动原 Master 节点（192.168.10.101）：
start-all.sh # 启动1个Master+2个Worker
启动备用 Master 节点（选择 192.168.10.102 作为备 Master）：

在192.168.10.102节点执行，仅启动Master进程

$SPARK_HOME/sbin/start-master.sh
查看 HA 集群状态：
访问原 Master Web UI：http://192.168.10.101:8080 → 状态为Active；
访问备 Master Web UI：http://192.168.10.102:8080 → 状态为Standby；
两个 Master 节点共用 2 个 Worker 节点，集群总资源不变。
1.6 验证 Master 主备切换（故障转移）
手动停止 Active Master 进程：

在192.168.10.101节点执行

jps # 查看Master进程号
kill -9 进程号 # 强制停止Active Master
验证切换结果：
访问备 Master Web UI：http://192.168.10.102:8080 → 状态变为Active；
Worker 节点自动连接新的 Active Master，无离线情况；
集群仍可正常接收和执行任务，实现无感知故障转移。
2. Spark On YARN 模式配置
Spark On YARN 模式是 Spark 整合 Hadoop 生态的核心模式，Spark 不再提供自己的集群管理，而是将资源调度交给 Hadoop YARN，Spark 作为客户端向 YARN 提交任务，由 YARN 分配节点资源运行任务。该模式是生产环境中大规模 Spark 集群的主流部署方式。
2.1 核心模式分类
YARN-client：Driver 程序运行在客户端机器，适用于开发调试，日志实时输出到客户端；
YARN-cluster：Driver 程序运行在 YARN 集群的 ApplicationMaster 节点，适用于生产环境，任务后台运行，稳定性更高。
2.2 配置 Spark 整合 YARN（所有节点）
复制 Hadoop 配置文件到 Spark 目录：让 Spark 识别 Hadoop/YARN 集群

进入Spark配置目录

cd $SPARK_HOME/conf

复制Hadoop核心配置文件

cp $HADOOP_HOME/etc/hadoop/core-site.xml .
cp $HADOOP_HOME/etc/hadoop/hdfs-site.xml .
cp $HADOOP_HOME/etc/hadoop/yarn-site.xml .
分发配置到所有节点：
scp $SPARK_HOME/conf/.xml root@192.168.10.102:$SPARK_HOME/conf/
scp $SPARK_HOME/conf/.xml root@192.168.10.103:$SPARK_HOME/conf/
启动 Hadoop 集群（HDFS+YARN）：

在Hadoop Master节点执行

start-dfs.sh
start-yarn.sh

查看YARN Web UI：http://192.168.10.101:8088 → 确认YARN集群正常运行

2.3 YARN-client 模式提交 Spark 任务

1. 将测试文件上传到HDFS（替代本地文件，实现分布式存储）

hdfs dfs -mkdir /input
hdfs dfs -put /usr/local/test.txt /input

2. 以YARN-client模式启动Spark Shell

spark-shell –master yarn –deploy-mode client

3. 执行HDFS上的WordCount任务

val textFile = sc.textFile(“hdfs:///input/test.txt”)
val wordCount = textFile.flatMap(.split(“ “)).map((,1)).reduceByKey(+)
wordCount.collect().foreach(println)

4. 查看任务：在YARN Web UI中可看到Spark任务运行记录

2.4 YARN-cluster 模式提交 Spark 任务
使用spark-submit提交打包好的 Spark 应用（以 Spark 官方示例包为例），适合生产环境：

提交WordCount示例任务到YARN-cluster模式

spark-submit
–master yarn
–deploy-mode cluster
–class org.apache.spark.examples.JavaWordCount
$SPARK_HOME/examples/jars/spark-examples_2.12-3.5.0.jar
hdfs:///input/test.txt

说明：

–class：指定应用的主类

最后一个参数：任务输入文件（HDFS路径）

任务提交后，可在YARN Web UI查看运行状态和结果

四、实验常见问题与解决
1Spark HA 启动后备 Master 为 Standby：属于正常现象，ZooKeeper 选举机制确保只有 1 个 Active Master，无需处理。
2.Spark On YARN 提交任务提示 HDFS 文件不存在：检查 HDFS 文件路径是否正确，Spark 配置目录是否有 Hadoop 的 core-site.xml 和 hdfs-site.xml。
3.YARN 集群资源不足：在 yarn-site.xml 中调整yarn.nodemanager.resource.memory-mb和yarn.nodemanager.resource.cpu-vcores，增大节点可用资源。
4.ZooKeeper 启动失败：检查 zoo.cfg 配置文件中 dataDir 和 dataLogDir 路径是否存在，节点间网络是否互通。
五、实验总结
本次实验完成了 Spark Standalone HA 模式和 Spark On YARN 模式的核心配置，解决了 Standalone 模式的单点故障问题：基于 ZooKeeper 实现了 Master 节点的主备切换，实现了集群高可用，保证了服务的持续运行。同时掌握了 Spark 与 Hadoop YARN 的整合方法，理解了 YARN-client 和 YARN-cluster 两种模式的差异和适用场景。
对比两种模式：Spark Standalone HA 模式部署简单，适用于中小型独立 Spark 集群；Spark On YARN 模式借助 Hadoop 的资源调度能力，实现了与 HDFS 的无缝整合，支持多租户资源分配，更适合生产环境的大规模、分布式 Spark 任务运行。本次实验也进一步理解了大数据生态中组件间的整合逻辑，为后续 Spark 高级应用开发和集群调优奠定了基础。
